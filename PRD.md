## 1. 背景与动机

当前 AI 伴侣 / 桌宠类产品大致可分为三类，每一类都各自存在体验缺口：

| 类型 | 代表方向 | 典型项目 | 主要短板 |
| --- | --- | --- | --- |
| 聊天机器人优先 | 多平台 IM Bot、Agent | AstrBot | 视觉存在感弱，仅是聊天框 |
| 虚拟角色框架优先 | AI VTuber、虚拟伙伴 | airi | 桌面常驻与桌宠交互不足 |
| 桌宠应用优先 | 桌面 Mate / 萌宠 | Mate-Engine | AI、插件、知识库扩展能力有限 |

**DeskSoul** 的定位是把这三类体验融合到同一个产品中：让 AI 角色像真正的"伙伴"一样常驻桌面，可以聊天、能驱动表情/动作、可扩展为 Agent 和 IM 生态的一部分，并通过角色包/插件机制由社区持续繁荣。

## 2. 产品定位

### 2.1 一句话定位
> **DeskSoul = 桌面常驻的 AI 角色伙伴。** 一个会被你看见、能和你聊天、可以被驱动出表情和动作、并允许你像装插件一样不断扩展能力的桌面灵魂体。

### 2.2 产品愿景
- **被看见**：角色不躲在聊天窗口里，而是在你桌面上"活着"。
- **能交流**：自然语言聊天、记得你、理解你的状态。
- **有反应**：AI 回复直接驱动表情、动作、台词、状态机。
- **可扩展**：通过角色包、插件、知识库、Agent 工具，让一个 DeskSoul 能成为很多种 DeskSoul。
- **属于社区**：核心开源 + 内容生态，让创作者能贡献角色与玩法。

### 2.3 核心价值主张
1. **存在感**：桌面级常驻，置顶 / 穿透 / 拖拽 / 互动，与窗口共处而非取代。
2. **对话即行为**：LLM 输出不仅是文本，更是 *动作 + 表情 + 状态变化* 的指令。
3. **角色感**：人格、记忆、口癖、关系递进，让对话有连贯人格。
4. **可拼装**：插件、Agent、知识库、IM Bridge 任意搭配。
5. **低心智成本**：开箱有默认角色、默认模型，可快速试用。

### 2.4 不做什么（Non-Goals，MVP 阶段）
- 不做完整的虚拟主播 / 直播推流套件。
- 不做底层模型训练，不预置自家闭源模型。
- 不做重度 3D 游戏化玩法（玩法插件可由社区做）。
- 不做强社交关系链 / 多人房间。

## 3. 目标用户

### 3.1 用户画像

| 画像 | 描述 | 主要诉求 |
| --- | --- | --- |
| **创作者 / 二次元爱好者** | 有自己的 OC、对 VRM/Live2D 有偏好 | 希望角色立绘能"动起来 + 会说话" |
| **AI 重度使用者** | 经常用 ChatGPT、Claude、本地 LLM | 想要一个常驻的 AI 入口，比浏览器更方便 |
| **效率向用户** | 程序员、写作者、设计师 | 桌面副驾驶：陪伴 + 提醒 + 简单 Agent 操作 |
| **插件/角色作者** | 有 Python/JS 基础 | 想做角色包、玩法、外接服务 |
| **轻度孤独陪伴用户** | 独居、远程工作 | 需要"有人在"的视觉与情感反馈 |

### 3.2 核心场景

- **S1 早安/晚安场景**：开机后角色自动迎接，根据时间和日历给出问候。
- **S2 工作陪伴场景**：长时间无操作时角色提醒休息；用户烦躁时表现安抚动作。
- **S3 主动唤起场景**：用户点击/呼唤角色，输入文字或语音问问题。
- **S4 情绪反馈场景**：AI 回复"开心 / 撒娇 / 害羞"等情绪标签 → 触发对应动作和表情。
- **S5 桌面 Agent 场景**：让角色帮忙搜索、总结剪贴板、调用本地工具。
- **S6 IM 同步场景（后期）**：在 QQ / Discord / Telegram 也能找到同一个角色。
- **S7 社区分享场景**：下载/订阅别人发布的角色包，立刻可用。

## 4. 竞品 / 参考项目分析

### 4.1 参考项目对比

| 维度 | Mate-Engine | AstrBot | airi | **DeskSoul（目标）** |
| --- | --- | --- | --- | --- |
| 核心定位 | 桌面虚拟伙伴（VRM 桌宠） | 多平台 IM 聊天机器人框架 | AI VTuber / 虚拟伙伴 | 桌面常驻 AI 角色伙伴 |
| 桌面常驻 | ✅ 强 | ❌ | ⚠️ 弱（偏 Web） | ✅ 强（一等公民） |
| 角色渲染 | VRM | 无 | VRM / Live2D | VRM + Live2D（双轨） |
| AI 对话 | 弱 | ✅ 多模型多平台 | ✅ | ✅（接 AstrBot 风格内核） |
| 表情/动作驱动 | 动画为主 | 无 | ✅（语音 + 动作） | ✅ AI 回复 → 行为 |
| 插件机制 | 有限 | ✅ 强（插件市场） | 模块化 | ✅（角色包 + 插件） |
| 知识库 / Agent | ❌ | ✅ | ⚠️ | ✅（V1+） |
| IM 生态 | ❌ | ✅ | ⚠️ | ✅（V2+，桥接 AstrBot） |
| 语音 / 嘴型 | 弱 | 无 | ✅ | ✅（V1+） |

### 4.2 复用策略
- **来自 Mate-Engine**：桌面窗口形态、置顶/穿透/拖拽交互、VRM 加载与基本动画体系，可直接借鉴或基于其思路在自家技术栈重写。
- **来自 AstrBot**：多 LLM Provider 抽象、插件系统、IM 适配层、知识库 / RAG，可作为 DeskSoul 的"AI 内核"或后端 Bridge。
- **来自 airi**：模块化角色 / 技能 / 记忆架构、语音 + 嘴型、Web 渲染思路，可借鉴模块边界设计。

> 原则：**优先体验，再考虑复用。** 不为了复用而拼接出割裂的产品形态。允许在 MVP 阶段不复用任何外部代码，只复用其架构经验；当某模块（如 IM Bridge）成熟项目能直接做后端时，再以"可选后端"的方式接入。

### 6.4 行为驱动（M4）

| ID | 需求 | 优先级 | 描述 / 验收 |
| --- | --- | --- | --- |
| F-BH-01 | 情绪标签解析 | P0 | LLM 输出可携带 `[emotion:happy]` 等标签或结构化 JSON |
| F-BH-02 | 标签 → 动作 | P0 | 情绪标签映射到动画 / 表情 / 持续时间 |
| F-BH-03 | 主动行为 | P1 | 长时间无交互时，角色主动 Idle 行为或开口 |
| F-BH-04 | 用户状态感知 | P2 | 系统忙/闲、专注模式联动（可选权限） |
| F-BH-05 | 行为脚本 | P1 | 角色包内可定义 `on_idle / on_click / on_message` 钩子 |

> 输出协议建议（最小集合）：
> ```json
> {
>   "text": "嗯…我在想要不要请你喝杯热可可",
>   "emotion": "shy",
>   "action": "fidget",
>   "duration_ms": 1800
> }
> ```
> 或在文本中内嵌不可见标签：`<emo:shy/>嗯…<act:fidget dur=1800/>我在想…`，由 EmotionEngine 解析。

## 7. 非功能性需求

| 类别 | 要求 |
| --- | --- |
| **平台** | Windows 10/11（首发）、macOS 12+（V1.0）、Linux X11/Wayland（V1.x） |

| **国际化** | 所有 UI 文案外置 i18n |
| **可观测性** | 本地日志环 + 用户主动反馈打包（脱敏） |

## 9. 关键交互与体验细节

1. **首次启动 90 秒体验**：开箱即得默认角色 → 引导填一个 LLM Key 或选择"试用本地模型" → 立刻能聊一句 → 引导触发一次表情驱动。
2. **沉默不打扰**：默认轻拟人化，避免"AI 助手感"满屏弹窗。所有主动行为可一键静音。
3. **AI 输出可视化**：流式输出与表情同步，最好让用户感觉"它一边想一边动"，而不是先停顿后整段播。
4. **角色 ≠ 模型**：同一个角色可以接不同 LLM，人格保持一致。
5. **失败优雅**：API 失败 / 超时不是错误弹窗，而是角色"歪头疑惑"+ 一句台词。
6. **隐私优先**：截屏 / 监听 / 屏幕内容相关能力默认关闭，开启需明确授权。

## 11. 复用与差异化策略

| 决策点 | 复用 | 自研 | 理由 |
| --- | --- | --- | --- |
| 桌宠窗口与交互 | 借鉴 Mate-Engine 思路 | 自研（Web 栈） | 与渲染层统一，便于跨平台 |
| 角色渲染 | three-vrm / pixi-live2d 现成库 | 仅做封装 | 避免重写引擎 |
| LLM / IM / 插件 | 长期 **可选接入 AstrBot** 作后端 | 自研轻量内核 | 桌面端必须有内置内核，AstrBot 作为"加倍器" |
| 模块化设计 | 借鉴 airi 的模块边界 | 自研 | 与桌宠形态结合度更高 |
| 角色包格式 | 参考 VRChat / VRM Avatar 元数据 | 自研 manifest | 给社区清晰协议 |

> **差异化总结**：桌面是一等公民、AI 输出直接驱动表现、角色包/插件双轨生态、内核可独立运行也可桥接成熟 IM 框架。

## 12. 风险与挑战

| 风险 | 影响 | 缓解策略 |
| --- | --- | --- |
| 跨平台透明窗口与穿透差异（尤其 Linux Wayland） | 高 | 优先 Win/Mac，Wayland 标记实验性 |
| LLM 成本与延迟 | 高 | 本地模型兜底、流式输出、Token 预算 |
| 表情驱动与文本流不同步导致违和 | 中 | 标签提前解析、动画过渡平滑 |
| 插件安全（恶意角色包） | 高 | 沙箱 + 权限提示 + 签名机制（V2） |
| 角色版权与素材合规 | 中 | 角色包元数据强制许可证字段；商店审核 |
| 隐私敏感（用户对"桌面常驻 AI"有顾虑） | 中 | 全部本地优先 + 明确开关 + 开源代码可审计 |
| 与现有 IM Bot 项目重复造轮子 | 中 | 通过 Bridge 复用 AstrBot，不抢生态 |

## 14. 附录

### 14.1 术语
- **Persona**：角色人格，含 System Prompt、口癖、关系设定。
- **角色包（Character Pack）**：模型 + 动画 + 人格 + 资源的可分发单元。
- **行为标签**：AI 输出中携带的可解析情绪 / 动作 / 状态。
- **Bridge 模式**：DeskSoul 内核作为前端，外接成熟后端（如 AstrBot）的运行模式。
